**Status:** ✅ Implemented & merged via PR

## Summary

Extended `email_automation.py` with two new public functions and one new trigger
template so that municipality-facing outreach artifacts include a concise,
demand-aware context block sourced from the municipality demand snapshot
(`compute_municipality_demand_signal()` from `insights_engine.py`, delivered
in US-001).

**Key design decisions:**

| Concept | Approach |
|---|---|
| Demand context fetching | `get_municipality_demand_context(bfs_number=None)` — thin wrapper around `compute_municipality_demand_signal()`; returns a flat dict keyed for direct template injection |
| Graceful degradation | `_empty_demand_context()` helper; any DB error or missing BFS falls back to `has_demand_data=False` so templates render safely without demand data |
| Outreach rendering | `render_municipality_outreach(municipality_name, recipient_email, bfs_number, tenant, app)` — composes subject, demand context, and Jinja2 HTML in one call |
| Template | New `templates/emails/municipality_outreach.html` with a conditional `demand-box` block; colour-coded by demand level (high/medium/low/none) |
| Trigger template registry | `TRIGGER_TEMPLATES["municipality_outreach"]` registered in `email_automation.py` alongside `formation_nudge` |
| Exception logging | Demand fetch errors are logged at WARNING level with `exc_info=True` before returning the empty context |
